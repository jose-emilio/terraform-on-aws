## 1.1 S3: Almacenamiento de Objetos de Clase Mundial

Amazon S3 es el servicio de almacenamiento de objetos más utilizado del mundo. Su propuesta de valor es radical: ofrece **11 nueves de durabilidad** (99.999999999%), escalabilidad ilimitada y consistencia fuerte de lectura tras escritura. No importa si subes un archivo y lo lees un milisegundo después — S3 garantiza que verás el dato más reciente.

> *"S3 no es solo un disco duro en la nube. Es la capa de datos más resiliente que existe: tus datos sobrevivirían la destrucción simultánea de dos centros de datos en la misma región. Y a diferencia de un disco duro, nunca se queda sin espacio."*

El recurso `aws_s3_bucket` en Terraform es deliberadamente simple — solo el nombre y las tags. Toda la configuración de seguridad, acceso y ciclo de vida va en recursos separados:

```hcl
resource "aws_s3_bucket" "data_lake" {
  bucket = "${var.project}-${var.environment}-data-lake"

  tags = {
    Name        = "${var.project}-data-lake"
    Environment = var.environment
    ManagedBy   = "terraform"
  }
}
```

**Unicidad global**: el nombre del bucket es único en todo AWS, para todos los clientes de todas las regiones. Si alguien ya tiene `mi-bucket`, no puedes usarlo. Usa siempre el patrón `${project}-${environment}-${purpose}` para minimizar colisiones.

## 1.2 Privacidad por Defecto: Block Public Access

Este es el recurso que ha evitado miles de brechas de datos. Antes de que existiera, era fácil crear accidentalmente un bucket públicamente accesible. Hoy, `aws_s3_bucket_public_access_block` es el primer recurso que debes añadir después de crear el bucket:

```hcl
resource "aws_s3_bucket_public_access_block" "data_lake" {
  bucket = aws_s3_bucket.data_lake.id

  block_public_acls       = true   # Bloquea ACLs públicas nuevas
  block_public_policy     = true   # Bloquea bucket policies que den acceso público
  ignore_public_acls      = true   # Ignora ACLs públicas existentes
  restrict_public_buckets = true   # Restringe acceso público incluso si hay policy
}
```

Los cuatro ajustes trabajan juntos como un candado redundante. Incluso si alguien añade por error una bucket policy que daría acceso público, `restrict_public_buckets = true` lo bloquea igualmente.

**En organizaciones**: se puede activar Block Public Access a nivel de cuenta de AWS, bloqueando el acceso público en todos los buckets de una sola operación. Esto es lo recomendado para cuentas de producción.

## 1.3 Control de Acceso: Bucket Policies

Las Bucket Policies son documentos JSON que definen permisos granulares sobre el bucket — quién puede hacer qué, desde dónde. Terraform usa `jsonencode()` para construirlas como estructuras HCL nativas, evitando strings escapados y permitiendo interpolación de variables:

```hcl
# Policy que deniega acceso si la petición NO viene de un VPC Endpoint
resource "aws_s3_bucket_policy" "vpc_only" {
  bucket = aws_s3_bucket.data_lake.id

  policy = jsonencode({
    Version = "2012-10-17"
    Statement = [{
      Sid       = "VPCOnly"
      Effect    = "Deny"
      Principal = "*"
      Action    = "s3:*"
      Resource  = [
        aws_s3_bucket.data_lake.arn,
        "${aws_s3_bucket.data_lake.arn}/*"
      ]
      Condition = {
        StringNotEquals = {
          "aws:sourceVpce" = aws_vpc_endpoint.s3.id
        }
      }
    }]
  })
}
```

Este patrón de "deny todo excepto desde la VPC" es seguridad perimetral a nivel de datos: aunque alguien filtrase las credenciales AWS de tu aplicación, no podría acceder al bucket desde fuera de la red privada.

Condiciones útiles en Bucket Policies:
- `aws:sourceVpce` — restringir al VPC Endpoint
- `aws:SecureTransport = true` — forzar HTTPS
- `aws:PrincipalOrgID` — limitar a miembros de tu AWS Organization
- `aws:IpAddress` — whitelist de IPs

## 1.4 Conectividad Privada: VPC Endpoints para S3

Por defecto, el tráfico desde EC2 a S3 sale a internet (aunque esté dentro de AWS). Los VPC Endpoints crean una ruta directa privada sin salir de la red de Amazon:

```hcl
# Gateway Endpoint: gratuito, basado en rutas
resource "aws_vpc_endpoint" "s3" {
  vpc_id            = aws_vpc.main.id
  service_name      = "com.amazonaws.${var.region}.s3"
  vpc_endpoint_type = "Gateway"

  tags = { Name = "${var.project}-s3-endpoint" }
}

# Añadir a la tabla de rutas privada
resource "aws_vpc_endpoint_route_table_association" "private" {
  vpc_endpoint_id = aws_vpc_endpoint.s3.id
  route_table_id  = aws_route_table.private.id
}
```

| Tipo | Coste | Mecanismo | On-premises |
|------|-------|-----------|-------------|
| **Gateway** | Gratuito | Tabla de rutas | No |
| **Interface** | Por hora + GB | ENI con IP privada (PrivateLink) | Sí (vía VPN/DX) |

Para la mayoría de los casos, el Gateway Endpoint es suficiente y gratuito. El Interface Endpoint solo es necesario si necesitas acceso desde on-premises o cross-region.

## 1.5 Resiliencia: Versionado de Objetos

El versionado protege contra dos de los escenarios más destructivos: borrado accidental y ransomware.

> *"Sin versionado, un script con un bug que borra objetos es irreversible. Con versionado, un Delete solo añade un 'Delete Marker' — el objeto real sigue ahí, recuperable en segundos."*

```hcl
resource "aws_s3_bucket_versioning" "data_lake" {
  bucket = aws_s3_bucket.data_lake.id

  versioning_configuration {
    status = "Enabled"   # También: "Suspended"
  }
}
```

**Tres estados**: `Enabled` (activo), `Suspended` (nuevos objetos sin versionar, versiones antiguas preservadas), y deshabilitado (solo al crear el bucket). **No se puede volver a deshabilitado una vez activado**.

**Coste**: cada versión ocupa espacio de almacenamiento. Combina siempre el versionado con Lifecycle Rules que expiren versiones antiguas (ej. mantener solo las últimas 30 versiones).

## 1.6 FinOps: Clases de Almacenamiento y Ciclo de Vida

S3 tiene varias clases de almacenamiento con distintos precios y latencias. Las Lifecycle Rules automatizan la transición entre ellas sin intervención manual:

```hcl
resource "aws_s3_bucket_lifecycle_configuration" "data_lake" {
  bucket = aws_s3_bucket.data_lake.id

  rule {
    id     = "archive-and-cleanup"
    status = "Enabled"

    filter { prefix = "logs/" }   # Solo aplica a esta "carpeta"

    # Paso 1: tras 30 días → Standard-IA (40% más barato)
    transition {
      days          = 30
      storage_class = "STANDARD_IA"
    }

    # Paso 2: tras 90 días → Glacier (80% más barato)
    transition {
      days          = 90
      storage_class = "GLACIER"
    }

    # Paso 3: borrar tras 1 año
    expiration { days = 365 }

    # Limpiar versiones antiguas tras 30 días
    noncurrent_version_expiration {
      noncurrent_days = 30
    }
  }
}
```

Para Data Lakes con patrones de acceso desconocidos, `Intelligent-Tiering` delega la decisión a ML de AWS:

```hcl
resource "aws_s3_bucket_intelligent_tiering_configuration" "full" {
  bucket = aws_s3_bucket.data_lake.id
  name   = "full-bucket-tiering"
  status = "Enabled"

  tiering {
    access_tier = "ARCHIVE_ACCESS"
    days        = 90   # Mover a Archive si no se accede en 90 días
  }

  tiering {
    access_tier = "DEEP_ARCHIVE_ACCESS"
    days        = 180   # Deep Archive tras 180 días sin acceso
  }
}
```

## 1.7 Inmutabilidad: Object Lock (WORM)

Para entornos regulados (financiero, salud, legal) donde los datos no pueden modificarse ni borrarse durante un período definido:

```hcl
resource "aws_s3_bucket" "compliance_vault" {
  bucket              = "${var.project}-compliance-vault"
  object_lock_enabled = true   # Debe activarse en la creación del bucket
}

resource "aws_s3_bucket_object_lock_configuration" "vault" {
  bucket = aws_s3_bucket.compliance_vault.id

  rule {
    default_retention {
      mode  = "COMPLIANCE"   # COMPLIANCE: ni root puede borrar
      years = 7              # Retención mínima de 7 años
    }
  }
}
```

| Modo | Quién puede eliminar | Caso de uso |
|------|---------------------|------------|
| **GOVERNANCE** | Solo usuarios con permiso `s3:BypassGovernance` | Testing, compliance flexible |
| **COMPLIANCE** | Nadie (ni root, ni AWS) hasta que expire | SEC, HIPAA, PCI-DSS |

## 1.8 Cifrado: SSE-KMS con Bucket Key

```hcl
resource "aws_s3_bucket_server_side_encryption_configuration" "kms" {
  bucket = aws_s3_bucket.data_lake.id

  rule {
    apply_server_side_encryption_by_default {
      sse_algorithm     = "aws:kms"
      kms_master_key_id = aws_kms_key.s3.arn
    }
    bucket_key_enabled = true   # Reduce llamadas a KMS en ~99% → ahorro significativo
  }
}
```

**`bucket_key_enabled = true`** es crítico para el coste: sin él, cada operación de cifrado/descifrado hace una llamada a KMS ($0.03 por 10,000 llamadas). Un bucket activo puede generar millones de llamadas al día. El Bucket Key usa una clave derivada en S3 que reduce las llamadas a KMS en hasta un 99%.

Tres modos de SSE:
- **SSE-S3**: AWS gestiona todo, sin coste extra, sin auditoría de acceso a claves — el default desde 2023
- **SSE-KMS**: CMK propio, auditoría en CloudTrail, rotación configurable — lo que necesitas en producción
- **SSE-C**: tú provees la clave en cada request, AWS no la almacena — máximo control, máxima complejidad

## 1.9 S3 Select y S3 Object Lambda

**S3 Select** permite ejecutar SQL directamente sobre objetos CSV, JSON o Parquet sin descargarlo completo:

> *"Imagina un archivo CSV de 10 GB con logs. Con S3 Select, filtras solo las líneas de los últimos 5 minutos con SQL, y AWS te devuelve 100 KB en lugar de 10 GB. Tu bandwidth y tu factura de transferencia te lo agradecerán."*

**S3 Object Lambda** intercepta peticiones GET y las procesa con una Lambda antes de devolver el objeto — redactar PII, redimensionar imágenes, convertir formatos — todo dinámicamente y transparente para el cliente.

## 1.10 Replicación: CRR y SRR

```hcl
resource "aws_s3_bucket_replication_configuration" "dr" {
  bucket = aws_s3_bucket.source.id
  role   = aws_iam_role.replication.arn

  rule {
    id     = "crr-all-objects"
    status = "Enabled"

    filter { prefix = "" }   # Replicar todo el bucket

    delete_marker_replication {
      status = "Enabled"   # Replicar también los deletes
    }

    destination {
      bucket        = aws_s3_bucket.replica.arn
      storage_class = "STANDARD_IA"   # Réplica más barata

      encryption_configuration {
        replica_kms_key_id = aws_kms_key.replica.arn   # Re-cifrar en destino
      }
    }

    source_selection_criteria {
      sse_kms_encrypted_objects { status = "Enabled" }
    }
  }
}
```

**Requisitos**: versionado habilitado en origen y destino, IAM Role con permisos de lectura en origen y escritura en destino, y solo replica objetos **nuevos** (los existentes hay que sincronizarlos con S3 Batch Operations).

| Tipo | Entre | Uso |
|------|-------|-----|
| **CRR** (Cross-Region) | Distintas regiones | DR, compliance, latencia global |
| **SRR** (Same-Region) | Misma región | Agregación de logs, dev→QA, soberanía de datos |

## 1.11 Access Points: Acceso Granular por Equipo

Los Access Points simplifican el control de acceso a data lakes compartidos. En lugar de una Bucket Policy compleja que gestione todos los equipos, cada equipo tiene su propio Access Point con su propia policy:

```hcl
resource "aws_s3_access_point" "analytics" {
  bucket = aws_s3_bucket.data.id
  name   = "analytics-team"

  vpc_configuration {
    vpc_id = aws_vpc.main.id   # Solo accesible desde esta VPC
  }

  policy = jsonencode({
    Version = "2012-10-17"
    Statement = [{
      Effect    = "Allow"
      Principal = { AWS = var.analytics_role }
      Action    = ["s3:GetObject"]
      Resource  = "*"
    }]
  })
}
```

El equipo de analytics usa el ARN del Access Point como endpoint — nunca ve el bucket directamente. Si necesitas revocar su acceso, borras el Access Point sin tocar el bucket.

## 1.12 Event Notifications: Reaccionar a Cambios en S3

```hcl
resource "aws_s3_bucket_notification" "events" {
  bucket      = aws_s3_bucket.data_lake.id
  eventbridge = true   # Enviar TODOS los eventos a EventBridge también

  # Notificar a SQS cuando se sube un CSV al prefijo uploads/
  queue {
    queue_arn     = aws_sqs_queue.processor.arn
    events        = ["s3:ObjectCreated:*"]
    filter_prefix = "uploads/"
    filter_suffix = ".csv"
  }
}
```

Dos modelos de integración:
- **Nativo S3** → SNS, SQS o Lambda: simple, filtros básicos por prefijo/sufijo
- **EventBridge**: filtros JSON avanzados, 18+ destinos distintos, archive & replay

Para pipelines de procesamiento de datos modernos, la combinación S3 + EventBridge + Step Functions o Lambda es el patrón más flexible.

## 1.13 Best Practices: el Módulo S3 Seguro

En una organización, el patrón correcto es encapsular todas las configuraciones de seguridad en un módulo que las aplique por defecto:

```hcl
module "secure_bucket" {
  source = "./modules/s3-secure"

  bucket_name       = "${var.project}-data-lake"
  kms_key_arn       = aws_kms_key.s3.arn
  enable_versioning = true
  enable_logging    = true
  log_bucket_id     = module.log_bucket.id

  lifecycle_rules = [{
    id              = "archive"
    transition_days = 90
    storage_class   = "GLACIER"
    expiration_days = 365
  }]

  tags = var.common_tags
}
```

El módulo garantiza que ningún bucket salga sin Block Public Access, sin SSE-KMS y sin versionado. Los equipos no pueden "olvidarse" de la seguridad porque está en la implementación del módulo.

## 1.14 Resumen: Amazon S3 con Terraform

| Componente | Recurso Terraform | Función |
|-----------|------------------|---------|
| Bucket | `aws_s3_bucket` | Contenedor de objetos |
| Privacidad | `aws_s3_bucket_public_access_block` | Bloquear acceso público |
| Políticas | `aws_s3_bucket_policy` + `jsonencode()` | Control granular de acceso |
| Conectividad | `aws_vpc_endpoint` (Gateway) | Tráfico privado sin internet |
| Versionado | `aws_s3_bucket_versioning` | Protección contra borrado |
| Lifecycle | `aws_s3_bucket_lifecycle_configuration` | Automatizar transiciones y expiración |
| Intelligent-Tiering | `aws_s3_bucket_intelligent_tiering_configuration` | ML para optimizar costes |
| Object Lock | `aws_s3_bucket_object_lock_configuration` | WORM para compliance |
| Cifrado | `aws_s3_bucket_server_side_encryption_configuration` | SSE-KMS + Bucket Key |
| Replicación | `aws_s3_bucket_replication_configuration` | CRR/SRR para DR |
| Access Points | `aws_s3_access_point` | Acceso granular por equipo/VPC |
| Notificaciones | `aws_s3_bucket_notification` | Integración event-driven |

> **Principio**: nunca crees un bucket sin Block Public Access + SSE-KMS + versionado. En producción, estos tres son no negociables. El resto — lifecycle rules, replicación, access points — depende del caso de uso.
